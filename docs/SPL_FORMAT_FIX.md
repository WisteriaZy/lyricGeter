# SPL 格式修正记录（2026-07-03）

## 修正内容

### 1. 逐字时间戳位置错误 ⭐

**问题**：时间戳放在字**之后**，不符合 SPL 标准。

❌ **错误格式**（旧实现）：
```
[00:13.86]ヤ[00:14.12]リ[00:14.30]タ[00:14.45]
     ↑字    ↑时间戳（错误：在字之后）
```

✅ **正确格式**（修正后）：
```
[00:13.86]ヤ[00:14.12]リ[00:14.30]タ[00:14.45]カ
     ↑时间戳  ↑字（正确：时间戳在字之前）
```

**SPL 标准原文**：
> 逐字歌词和行歌词的时间戳逻辑差别不大，如上方提到的需求可以写成：
> ```
> [05:20.22]你好[05:23.22]椒盐音乐[05:24.22]
> ```
> 其中 `[05:20.22]` 依旧是行开始时间戳同时也是开始逐字标记时间戳，因为它位于句首，而 `[05:23.22]` 是中间的一个逐字标记时间戳，用以分隔 `你好` 和 `椒盐音乐`...

### 2. 行结束时间戳缺失

**问题**：网易云逐字格式转换时，部分情况下缺少行结束时间戳。

✅ **修正逻辑**：
- 有下一行时：使用下一行开始时间作为当前行结束时间
- 最后一行时：使用 `行起始时间 + 持续时间` 作为结束时间

**代码修改**（`converter.py`）：
```python
# 修正前
parts.append(word_text)
parts.append(_ms_to_stamp(word_end))  # ❌ 错误：时间戳在字后

# 修正后
parts.append(_ms_to_stamp(word_time))  # ✅ 正确：时间戳在字前
parts.append(word_text)

# 行结束时间戳（新增）
if i + 1 < len(lines):
    parts.append(_ms_to_stamp(lines[i + 1].start))
else:
    parts.append(_ms_to_stamp(line.start + line.duration))
```

### 3. 时间戳格式说明

**保持 2 位毫秒**（厘秒精度，10ms），符合 SPL 规范：

```python
def _ms_to_stamp(ms: int) -> str:
    """毫秒 → [mm:ss.xx] 格式。毫秒部分：最少1位、最多6位，通常保留2位（10ms精度）。"""
    m, rem = divmod(ms, 60_000)
    s, ms_part = divmod(rem, 1_000)
    cs = ms_part // 10  # 厘秒（centisecond）
    return f"[{m:02d}:{s:02d}.{cs:02d}]"
```

**SPL 标准原文**：
> 毫秒限制 1 至 6 位数字，如 `1`、`02`、`130`、`450000` 都是支持的写法。不足 3 位的写法将视为在后位省略了 `0`，如 `1` 将视为 `100`、`02` 将视为 `020`，分别对应 100 毫秒和 20 毫秒，而非 1 毫秒和 2 毫秒。

## 延迟逐字标记（未实现）

SPL 标准支持延迟逐字标记（用 `<>` 包裹），实现行到达但首字未开始的效果：

```spl
[05:20.22]<05:21.22>你好<05:23.22>椒盐音乐[05:24.22]
    ↑ 行开始    ↑ 首字开始
```

**当前状态**：网易云逐字格式不包含此特性，暂不实现。

**未来扩展**：如果其他平台（酷狗 KRC、QQ 音乐 QRC）支持延迟逐字，可在 `converter.py` 中添加转换逻辑。
